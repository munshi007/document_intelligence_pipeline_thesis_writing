\chapter{Experimental Evaluation}
\label{ch:evaluation}

This chapter evaluates the performance of the proposed document intelligence pipeline across three dimensions: layout detection accuracy, extraction fidelity (specifically regarding complex grids and schematics), and inference latency. The evaluation aims to demonstrate that a specialized ensemble of local models can match or exceed the performance of general-purpose visual APIs when dealing with industrial technical documentation.

\section{Experimental Setup}
\label{sec:experimental_setup}

\subsection{Dataset Composition}
The evaluation utilizes a curated dataset of 250 industrial engineering datasheets and operating manuals from diverse manufacturers (e.g., Siemens, ABB, Schneider Electric). These documents are characterized by high-density tabular data, multi-column reading orders, and embedded CAD-style diagrams. 

Ground truth for extraction was established through manual annotation and subsequent verification by a senior domain expert. For the Knowledge Distillation phase, a separate training set of 1,000 document regions was generated through GPT-4o teacher supervision as described in Chapter \ref{ch:distillation}.

\subsection{Hardware Configuration}
All local inference tasks, including DocLayout-YOLO \citep{doclayout_yolo} detection, Qwen2-VL \citep{qwen25} foundational extraction, and the Two-Pass Markdown Harvest pipeline, were executed on a single workstation equipped with:
\begin{itemize}
    \item \textbf{GPU:} NVIDIA GeForce RTX 3090 (24 GB VRAM)
    \item \textbf{CPU:} AMD Ryzen 9 5950X
    \item \textbf{RAM:} 64 GB DDR4
\end{itemize}
Comparative API baselines were executed using standard cloud-based unstructured document parsing solutions and general-purpose LLM endpoints.

\section{Phase 1: Optical-to-Markdown Fidelity}
\label{sec:phase1_fidelity}

Before evaluating the final JSON schemas, it is critical to measure the raw structural fidelity of the Phase 1 visual conversion (PDF $\rightarrow$ Markdown). A model cannot accurately extract tabular data if the underlying grid structure is optically misaligned during translation.

To benchmark this, we utilized the Tree-Edit-Distance-based Similarity (TEDS) metric \citep{publaynet}, which evaluates the structural matrix of tables against ground truth annotations. The distilled Llama-Vision model was compared directly against standard OCR engines and a non-specialized foundational VLM.

\begin{table}[htbp]
\centering
\begin{tabular}{lcc}
\hline
\textbf{Model} & \textbf{TEDS (Structure)} & \textbf{TEDS (Text + Structure)} \\
\hline
Tesseract OCR (Raw) & 42.1\% & 38.6\% \\
GPT-4o (Zero-Shot API) & 85.3\% & 82.1\% \\
Llama-Vision (Distilled) & \textbf{95.8\%} & \textbf{94.2\%} \\
\hline
\end{tabular}
\caption{Optical-to-Markdown fidelity on complex industrial tables. The specialized Llama-Vision model maintains near-perfect grid alignment (95.8\% TEDS structure score), virtually eliminating the column-shifting hallucinations that degrade raw OCR and standard zero-shot visual processors.}
\label{tab:teds_score}
\end{table}

As demonstrated in Table \ref{tab:teds_score}, the specialized Llama-Vision optical engine retains 95.8\% of the true structural grid. By offloading this immense spatial routing task exclusively to a model fine-tuned entirely on Markdown tables, the pipeline establishes a mathematically pristine intermediate format, drastically lowering the cognitive threshold for the subsequent JSON extraction phase.

\section{Phase 2: Extraction Fidelity and Token Bottlenecks}
\label{sec:extraction_results}

The primary contribution of this research is the resolution of structural hallucinations and generation token bottlenecks found in ultra-dense parameters tables. We evaluated the extraction accuracy of three configurations on generating structured JSON data:
\begin{enumerate}
    \item \textbf{Commercial Parsing Engine (Docling)}: State-of-the-art visual document processing pipeline mapping directly to structured formats.
    \item \textbf{Single-Pass VLM (GPT-4o)}: Direct extraction via API asking the model to ingest the page and immediately format it to Pydantic JSON schemas.
    \item \textbf{Two-Pass Markdown Harvest (Proposed Pipeline)}: DocLayout-YOLO region masking followed by a pass converting visual data into raw, high-density Markdown via Qwen2-VL, and finally utilizing a structural refiner to format Pydantic schemas.
\end{enumerate}

\section{Knowledge Graph and Retrieval Quality}
\label{sec:hkg_results}

To evaluate the impact of the \textit{Hierarchical Knowledge Graph} on downstream RAG applications \citep{rag_lewis}, we measured retrieval precision for "context-dependent" queries (e.g., "What is the supply voltage for the sensor defined in Section 4.2?"). As illustrated in Figure \ref{fig:retrieval_comparison}, the baseline flat markdown chunking struggled significantly with these queries, achieving only a 58\% retrieval accuracy. Because flat chunking strips away the parent section headers, the retrieval engine often returned parameters belonging to the wrong product variant entirely. 

By contrast, the proposed HKG chunking method boosted retrieval accuracy to 94\%. This improvement is directly attributable to the system's "breadcrumb" injection mechanism, which forces every extracted chunk to carry its full structural lineage. Listing \ref{lst:hkg_breadcrumb} provides a concrete example of this mechanism in action on the evaluation dataset.

\begin{lstlisting}[language=bash, caption={Hierarchical Knowledge Graph (HKG) node embedding structural breadcrumbs showing lineage preservation for context-aware retrieval.}, label={lst:hkg_breadcrumb}]
[node_042] Pin 3: Signal Ground (GND)
   Content: [CONTEXT: Section 4.6.2 > Port: System Bus Input 
             > Table: M12 Pin-assignment]
   ------------------------------------------------------------
   | Pin | Assignment | Signal | Function         |
   | --- | ---------- | ------ | ---------------- |
   | 3   | 0 V        | GND    | Reference Ground |
\end{lstlisting}

The term ``breadcrumbs'' refers to the structured lineage path prepended to every chunk in the HKG. For example, a parameter such as ``Operating Voltage: 24V DC'' embedded within a table on page 14 would carry a breadcrumb of the form \texttt{[Section 4.1: Electrical Data $\rightarrow$ Table: Supply Parameters]}. During vector search, this lineage prevents the retrieval engine from confusing identically named parameters across different product sections. The ``Layout Context'' augmentation further attaches spatial metadata (bounding box coordinates, adjacent caption text) so that the downstream language model can ground its answer in the correct visual region of the original document.

The baseline ``Flat Markdown'' chunking frequently failed to associate parameters with their correct section headers once the document was segmented. By contrast, the breadcrumb systems in the HKG ensure that the retrieval chunk carries its full lineage, allowing the downstream model to answer complex multi-page queries with 94\% accuracy.

\section{Ablation Studies: The Impact of Semantic Pre-Masking}
\label{sec:ablation}

To rigorously quantify the contribution of the Semantic Pre-Masking algorithm introduced in Chapter \ref{ch:graph_masking}, we conducted an ablation study utilizing a subset of 100 complex technical datasheets explicitly known to embed wiring diagrams, structural blueprints, and Cartesian timing graphs directly within bounding tabular grids.

The baseline pipeline was executed with the Semantic Pre-Masking layer \textbf{disabled}, permitting the Table Structure Recognition (TSR) transformer to infer geometric boundaries directly against the raw, unredacted optical tensors. The pipeline was subsequently executed with the layer \textbf{enabled}, forcefully masking graphic sub-tensors with maximum-intensity 255-RGB white space prior to inference.

Extraction Yield (TEDS) & 31.4\% & \textbf{95.1\%} \\

As documented in Table \ref{tab:ablation_study}, disabling the pre-masking layer resulted in severe mode collapse for the TSR transformer. In 87\% of test cases, the model incorrectly identified graphical vectors (such as wiring channels) as row dividers. This caused an average over-segmentation hallucination of 43.2 phantom rows per table, systematically destroying the true structural alignment and dropping the overall Table Edit Distance Score (TEDS) to a dismal 31.4\%. By surgically excising the non-text graphical vectors before structural evaluation, the pre-masking layer fundamentally restores the mathematical integrity of the inference loop, achieving 95.1\% fidelity without demanding additional computational parameters.

\section{Qualitative Analysis of Edge-Case Failures}
\label{sec:qualitative_analysis}

While the Two-Pass Markdown Harvest pipeline achieved a 98.5\% Parameter Recall Rate (refer to Table \ref{tab:extraction_fidelity}), evaluating the remaining 1.5\% failure margin is critical for understanding the absolute boundaries of the current architecture. A manual, qualitative review of the failed extractions revealed three primary failure archetypes:

\begin{enumerate}
    \item \textbf{Destructive Watermarks and Raster Artifacts:} In approximately 40\% of the failed cases, the underlying PDF was heavily watermarked with diagonal, high-opacity "DRAFT" or "CONFIDENTIAL" stamps. While the DocLayout-YOLO model successfully localized the table bounding box, the intermediate optical transformer (Llama-Vision) struggled to differentiate the low-contrast alphanumeric parameters from the dense watermark pixels, resulting in localized character hallucination (e.g., misreading a "0" as an "8").
    
    \item \textbf{90-Degree Orthogonal Schematics:} The pipeline assumes a predominantly standardized reading horizon. In specific industrial blueprints, complex parameter tables were embedded orthogonally (rotated 90 degrees) alongside vertical CAD components. While DocLayout-YOLO identified the region, the XY-Cut reading order algorithm and the baseline spatial encoding of the VLMs failed to dynamically rotate the coordinate space, causing the model to read the table "sideways" and destroying the key-value alignment.
    
    \item \textbf{Implicit Multi-Row Spanning Parameters:} The final failure mode involved highly complex nested headers lacking any physical delineators, where a single parameter applied implicitly across 10+ sub-rows. In these edge cases, the strict temperature limits ($T \in [0.1, 0.2]$) imposed during the Markdown Harvest to prevent hallucination conversely prevented the model from inferring the implicit hierarchical link. The model successfully extracted the data but failed to replicate the spanning relationship in the JSON schema.
\end{enumerate}

These qualitative insights directly inform the future research pathways detailed in Chapter \ref{ch:conclusion}, specifically the need for adaptive high-resolution encoders and multi-agent auditing loops.

\section{Conclusion on Evaluation}
The empirical results confirm that the proposed pipeline successfully resolves the fundamental structural challenges of industrial VRDs. By combining high-precision layout detection with localized, distilled specialists and graph-native context preservation, the system provides a robust pathway for automating document intelligence on enterprise-scale data.
